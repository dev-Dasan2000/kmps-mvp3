1. Employee Management

Base Path: /employees

Get All Employees

GET /employees

Retrieves a list of all employees with their bank information and emergency contacts.

Response Format:

[ { "eid": 1, "name": "John Doe", "dob": "1990-01-01", "gender": "male", "email": "john.doe@example.com", "phone": "1234567890", "address": "123 Main St", "city": "New York", "state": "NY", "province": null, "zip_code": "10001", "job_title": "Software Engineer", "employment_status": "full time", "salary": 75000, "bank_info": [ { "eid": 1, "account_holder": "John Doe", "account_no": "1234567890", "bank_name": "ABC Bank", "branch": "Main Branch", "account_type": "Savings" } ], "emergency_contact": [ { "eid": 1, "name": "Jane Doe", "relationship": "Spouse", "phone": "0987654321", "email": "jane.doe@example.com" } ] } ]

Get Employee by ID

GET /employees/:id

Retrieves a specific employee by their ID.

URL Parameters:

  • id - Employee ID (integer)

Response:

Returns single employee object with same structure as Get All Employees

Error Responses:

  • 404 Not Found: Employee with the specified ID does not exist

Create Employee

POST /employees

Creates a new employee record.

Required Fields:

  • name - Employee full name
  • dob - Date of birth (YYYY-MM-DD format)
  • gender - "male" or "female"
  • email - Valid email address
  • phone - Contact number
  • employment_status - "part time" or "full time"

Request Body Example:

{ "name": "John Doe", "dob": "1990-01-01", "gender": "male", "email": "john.doe@example.com", "phone": "1234567890", "address": "123 Main St", "city": "New York", "state": "NY", "province": "", "zip_code": "10001", "job_title": "Software Engineer", "employment_status": "full time", "salary": 75000, "bank_info": [ { "account_holder": "John Doe", "account_no": "1234567890", "bank_name": "ABC Bank", "branch": "Main Branch", "account_type": "Savings" } ], "emergency_contact": [ { "name": "Jane Doe", "relationship": "Spouse", "phone": "0987654321", "email": "jane.doe@example.com" } ] }

Error Responses:

  • 400 Bad Request: Missing required fields or validation errors

Update Employee

PUT /employees/:id

Updates an existing employee's information.

URL Parameters:

  • id - Employee ID (integer)

Request Body:

Same as Create Employee (partial updates allowed)

Error Responses:

  • 404 Not Found: Employee with the specified ID does not exist

Delete Employee

DELETE /employees/:id

Deletes an employee record.

URL Parameters:

  • id - Employee ID (integer)

Success Response:

{ "message": "Employee deleted successfully" }

Error Responses:

  • 404 Not Found: Employee with the specified ID does not exist

Bank Information Management

Get Bank Information

GET /employees/:id/bank-info

Retrieves all bank accounts for a specific employee.

URL Parameters:

  • id - Employee ID (integer)

POST /employees/:id/bank-info

Adds a new bank account for an employee.

Required Fields:

  • account_holder - Account holder name
  • account_no - Account number
  • bank_name - Bank name

Update Bank Information

PUT /employees/:id/bank-info/:account_no

Updates an existing bank account.

URL Parameters:

  • id - Employee ID (integer)
  • account_no - Account number (string)

Delete Bank Information

DELETE /employees/:id/bank-info/:account_no

Deletes a bank account.

URL Parameters:

  • id - Employee ID (integer)
  • account_no - Account number (string)

Emergency Contact Management

Get Emergency Contacts

GET /employees/:id/emergency-contacts

Retrieves all emergency contacts for a specific employee.

Add Emergency Contact

2. Attendance Management

Base Path: /attendance

Get Employee Attendance

GET /attendance/:eid

Retrieves attendance records for a specific employee for the current month.

URL Parameters:

  • eid - Employee ID (integer)

Response Example:

[ { "eid": 1, "name": "John Doe", "date": "2025-07-01", "clock_in": "09:00 AM", "clock_out": "05:00 PM" } ]

Get Total Attendance Days

GET /attendance/total/:eid

Retrieves the total attendance days for a specific employee, including approved leave days.

Response Example:

{ "eid": 1, "name": "John Doe", "total_attendance_days": 20, "total_leave_days": 5, "effective_attendance": 25 }

Get Attendance with Leave Information

GET /attendance/with-leaves/:eid

Retrieves attendance and leave information for a specific employee.

Response Example:

{ "eid": 1, "name": "John Doe", "total_days_present": 20, "total_leave_days": 5, "effective_attendance": 25, "leaves": [ { "from_date": "2025-07-10", "to_date": "2025-07-15", "type": "Annual Leave", "days": 6 } ] }

Get Weekly Attendance

GET /attendance/weekly/all

Retrieves weekly attendance for all employees, including leave information.

Response Structure:

Returns an array of objects containing weekly attendance data for each employee, including:

  • Daily attendance records
  • Leave information per day
  • Total days present and on leave
  • Week date range

Record Clock In

POST /attendance/clock-in

Records a clock-in event for an employee.

Request Body:

{ "eid": 1 }

Success Response:

{ "message": "Clock in recorded successfully", "attendance": { "eid": 1, "name": "John Doe", "clock_in": "09:00 AM", "date": "2025-07-07" } }

Error Responses:

  • 404 Not Found: Employee with the specified ID does not exist
  • 400 Bad Request: Employee is already clocked in today

Record Clock Out

POST /attendance/clock-out

Records a clock-out event for an employee.

Request Body:

{ "eid": 1 }

Error Responses:

  • 404 Not Found: Employee with the specified ID does not exist
  • 400 Bad Request: No active clock in found for today

3. Leave Management

Base Path: /leaves

Get All Leave Requests

GET /leaves

Retrieves all leave requests across all employees.

Response Example:

[ { "eid": 1, "employee_name": "John Doe", "from_date": "2025-07-10", "to_date": "2025-07-15", "type": "Annual Leave", "status": "Approved", "job_title": "Software Engineer", "employment_status": "full time" } ]

Get Leaves for Specific Employee

GET /leaves/:eid

Retrieves all leave requests for a specific employee.

URL Parameters:

  • eid - Employee ID (integer)

Response Example:

[ { "eid": 1, "employee_name": "John Doe", "from_date": "2025-07-10", "to_date": "2025-07-15", "type": "Annual Leave", "status": "Approved", "duration": 6 } ]

Error Responses:

  • 404 Not Found: Employee with the specified ID does not exist

Apply for Leave

POST /leaves

Creates a new leave request.

Required Fields:

  • eid - Employee ID (integer)
  • from_date - Start date (YYYY-MM-DD format)
  • to_date - End date (YYYY-MM-DD format)
  • type - Leave type (e.g., "Annual Leave", "Sick Leave")

Request Body Example:

{ "eid": 1, "from_date": "2025-07-10", "to_date": "2025-07-15", "type": "Annual Leave" }

Success Response:

{ "message": "Leave request submitted successfully", "leave": { "eid": 1, "from_date": "2025-07-10", "to_date": "2025-07-15", "type": "Annual Leave", "status": "Pending" } }

Error Responses:

  • 404 Not Found: Employee with the specified ID does not exist
  • 400 Bad Request: End date cannot be before start date
  • 409 Conflict: Overlapping leave request exists

Update Leave Status

PUT /leaves/:eid/:fromDate/:toDate/status

Updates the status of a leave request (approve, reject, or cancel).

URL Parameters:

  • eid - Employee ID (integer)
  • fromDate - Start date (YYYY-MM-DD format)
  • toDate - End date (YYYY-MM-DD format)

Request Body:

{ "status": "Approved" }

Valid Status Values:

  • "Approved"
  • "Rejected"
  • "Cancelled"

Success Response:

{ "message": "Leave request approved successfully", "leave": { "eid": 1, "from_date": "2025-07-10", "to_date": "2025-07-15", "type": "Annual Leave", "status": "Approved" } }

Error Responses:

  • 404 Not Found: Leave request not found
  • 400 Bad Request: Invalid status value

Delete Leave Request

DELETE /leaves/:eid/:fromDate/:toDate

Deletes a leave request.

URL Parameters:

  • eid - Employee ID (integer)
  • fromDate - Start date (YYYY-MM-DD format)
  • toDate - End date (YYYY-MM-DD format)

Success Response:

{ "message": "Leave request deleted successfully" }

Error Responses:

  • 404 Not Found: Leave request not found

Get Leave Summary

GET /leaves/summary/:eid

Retrieves a summary of approved leaves by type for a specific employee.

URL Parameters:

  • eid - Employee ID (integer)

Response Example:

{ "eid": 1, "name": "John Doe", "summary": { "Annual Leave": { "total_days": 15, "count": 2 }, "Sick Leave": { "total_days": 5, "count": 1 } } }

Error Responses:

  • 404 Not Found: Employee with the specified ID does not exist

Integration Notes for Frontend Developers

Authentication & Headers

All API requests should include appropriate headers. While not explicitly documented, ensure you include:

  • Content-Type: application/json for POST/PUT requests
  • Any authentication tokens as required by your system

Error Handling

Always implement proper error handling for HTTP status codes:

  • 200 OK: Successful GET requests
  • 201 Created: Successful POST requests
  • 400 Bad Request: Invalid request data
  • 404 Not Found: Resource not found
  • 409 Conflict: Data conflicts (e.g., overlapping leave requests)
  • 500 Internal Server Error: Server-side errors

Date Format Standards

All dates in the API follow these formats:

  • Date fields: YYYY-MM-DD (e.g., "2025-07-10")
  • Time fields: HH:MM AM/PM (e.g., "09:00 AM")

Data Validation

Client-side validation should match server-side requirements:

  • Gender: Must be "male" or "female"
  • Employment Status: Must be "part time" or "full time"
  • Leave Status: Must be "Pending", "Approved", "Rejected", or "Cancelled"
  • Email: Must be a valid email format
  • Phone: Should be numeric string

Common Use Cases

Employee Registration Flow:

  1. Create employee with basic information
  2. Add bank information (if provided)
  3. Add emergency contacts (if provided)

Daily Attendance Flow:

  1. Employee clocks in using /attendance/clock-in
  2. Employee clocks out using /attendance/clock-out
  3. View attendance records using /attendance/:eid

Leave Management Flow:

  1. Employee applies for leave using /leaves
  2. Manager views all leave requests using /leaves
  3. Manager approves/rejects using /leaves/:eid/:fromDate/:toDate/status
  4. View leave summary using /leaves/summary/:eid

Performance Considerations

  • Use pagination for large employee lists (if supported)
  • Cache employee data when possible
  • Implement loading states for API calls
  • Use debouncing for search functionality

Testing Recommendations

  • Test all CRUD operations for each endpoint
  • Test error scenarios (404, 400, 409)
  • Test date edge cases (leap years, month boundaries)
  • Test concurrent clock-in/clock-out scenarios
  • Test overlapping leave requests
